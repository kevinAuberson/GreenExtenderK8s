# carbon-signal

Carbon signal aggregator. Polls Electricity Maps (grid carbon intensity),
vCenter (real per-host power via ESXi sensors), and the Kubernetes
metrics-server (per-node CPU/memory load), then publishes the combined
result to the `carbon-signal` ConfigMap consumed by the extender.

## How it works

![Architecture overview](../docs/architecture.png)

This service runs the **collection cycle** (solid arrows): every
`POLL_INTERVAL` seconds it polls vSphere, metrics-server, and Electricity
Maps independently (each with its own try/except, so one source failing
doesn't block the others), merges the results in memory, and writes them
to the `carbon-signal` ConfigMap. This cycle is continuous and completely
decoupled from pod scheduling — the extender only ever reads that
ConfigMap, never these sources directly.

## Environment variables

Copy `.env.example` to `.env` for local runs (`load_dotenv()` in `main.py`
picks it up automatically). In the cluster, these are set via
`manifests/aggregator/aggregator-deployment.yaml` (non-sensitive) and the
`carbon-signal-secrets` Secret (sensitive).

### Required

| Variable | Used by | Description |
|---|---|---|
| `EMAPS_TOKEN` | `electricity_maps.py` | Electricity Maps API token. Get one at https://api-portal.electricitymaps.com/ |
| `VCENTER_HOST` | `vsphere.py` | vCenter hostname/IP |
| `VCENTER_USER` | `vsphere.py` | vCenter service account username |
| `VCENTER_PASSWORD` | `vsphere.py` | vCenter service account password |

Missing any of these raises `KeyError` at startup — there is no silent
fallback, by design.

### Optional

| Variable | Default | Used by | Description |
|---|---|---|---|
| `EMAPS_ZONE` | `CH` | `electricity_maps.py` | Electricity Maps zone code to query |
| `EMAPS_CURRENT_TTL` | `300` | `electricity_maps.py` | Cache TTL (s) for current carbon intensity. Tune to your API plan: academic/unlimited → `300`, commercial (~1000 req/day) → `120`, free (~100 req/day) → `900` |
| `EMAPS_FORECAST_TTL` | `3600` | `electricity_maps.py` | Cache TTL (s) for the 24h forecast. Free tier → `86400` |
| `POLL_INTERVAL` | `30` | `main.py` | Seconds between aggregator polling cycles. Should track `EMAPS_CURRENT_TTL` so you don't poll faster than the data actually refreshes |
| `THRESHOLDS_FILE` | `/app/thresholds.yaml` | `main.py` | Path to the green/dirty threshold YAML, mounted from the `carbon-signal-thresholds` ConfigMap (generated by `scripts/compute_thresholds.py`) |
| `IN_CLUSTER` | `false` | `main.py`, `metrics_server.py` | Set to `"true"` only when running inside a Kubernetes pod — switches from local `~/.kube/config` to the in-cluster service account config |

### Reserved (currently unused)

These are present in `.env.example` and the deployment manifest, but no
code path reads them today. Documented here so they aren't mistaken for
a working knob.

| Variable | Notes |
|---|---|
| `PROMETHEUS_URL` | Node telemetry currently comes from vSphere (power) and metrics-server (CPU/RAM), not a direct Prometheus query — nothing in `carbon-signal/` reads this variable. |
| `VCENTER_VERIFY_SSL` | TLS verification against vCenter is hardcoded off in `vsphere.py` (self-signed cert, network isolated by firewall) regardless of this variable's value. |

## Local run

```bash
cp .env.example .env    # fill in your values
pip install -r requirements-dev.txt
python main.py
```
